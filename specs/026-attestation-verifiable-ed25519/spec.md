# Feature Specification: F08 — Attestation Vérifiable Ed25519 + QR + Page Publique `/verify/{id}` + Révocation (Module 5.3)

**Feature Branch**: `feat/F08-attestation-verifiable-ed25519` (alias SpecKit `026-attestation-verifiable-ed25519`)
**Created**: 2026-05-07
**Status**: Draft
**Input**: User description : « F08 — Attestation Vérifiable Ed25519 + QR + Page publique `/verify/{id}` + Révocation. Une PME génère une attestation PDF de son score crédit + ESG signée numériquement Ed25519 avec QR code pointant vers une page publique sans authentification (`/verify/{attestation_id}`) qu'un fund officer scanne pour vérifier l'authenticité hors-plateforme. Création d'un modèle `Attestation` séparé de `credit_scores` (signature détachée payload + hash PDF, public_key_id versionné, validité 1 an, révocation par PME ou admin avec audit log F03), endpoint backend public no-auth `GET /api/public/verify/{id}`, page Vue `pages/verify/[id].vue` (no-auth via middleware), layout `public.vue` réutilisable, refactor du tool LLM `generate_credit_certificate` pour appel réel au service. »

## Contexte produit

Le brainstorming Module 5.3 (Innovation 9) identifie l'attestation vérifiable comme **différenciateur produit n°2** : « la plateforme étant fermée aux intermédiaires, le partage du score se fait via une attestation vérifiable que la PME contrôle et transmet par ses propres canaux. Le fund officer scanne un QR code pour vérifier l'authenticité — pas besoin de compte. » L'état actuel ne tient pas cette promesse. Le module crédit dispose d'un PDF basique (`backend/app/modules/credit/certificate.py`) sans signature numérique, sans QR code, sans identifiant unique exposé publiquement, et le tool LLM `generate_credit_certificate` est un placeholder qui retourne une URL fictive sans appeler le service de génération. Aucune page publique `/verify/{id}` n'existe (le middleware `auth.global.ts` n'autorise actuellement que `/login` et `/register` en pages publiques). Aucune logique de révocation n'est implémentée. Tant que F08 n'est pas livrée, l'attestation n'est **techniquement pas vérifiable** : un PDF peut être altéré, l'URL fictive ne mène nulle part, et aucun canal hors-plateforme ne permet à un partenaire fonds de valider la chaîne d'attestation. Le risque crédibilité est majeur : c'est cette feature qui transforme la plateforme d'« outil interne » en « tiers de confiance reconnu par les fonds verts ».

F08 introduit donc : (1) un modèle `Attestation` premier rang multi-tenant (F02), Auditable (F03), avec signature Ed25519 détachée du couple `payload + hash PDF` ; (2) la génération de QR code pointant vers `https://esg-mefali.com/verify/{attestation_id}` ; (3) un endpoint backend public no-auth `GET /api/public/verify/{id}` exposant un statut whitelisté (authentic/revoked/expired/invalid) sans fuite d'information ; (4) un layout Nuxt `public.vue` et une page `pages/verify/[id].vue` accessibles sans authentification (modification du middleware global) ; (5) la révocation par la PME elle-même ou par un admin avec audit log ; (6) le refactor du tool LLM `generate_credit_certificate` pour appel réel au service ; (7) un template HTML enrichi affichant le QR, le hash SHA-256 du PDF en pied de page, et l'annexe sources auto-générée (F01). La couverture de test cible ≥ 85 % sur les couches `signing`, `qr` et `service`.

## Clarifications

> **Mode autonomie totale** — l'utilisateur est absent. Pour chaque question, le choix « recommandé » a été appliqué selon les invariants ESG Mefali, la stack imposée et la fiche `documents_et_brouillons/features_a_implementer/F08-attestation-verifiable-ed25519.md`. Chaque décision est tracée ci-dessous avec son rationale.

### Session 2026-05-07

- **Q : Quelle bibliothèque cryptographique pour la signature Ed25519 — `cryptography>=41.0` ou `pynacl>=1.5` ?**
  → **A : `cryptography>=41.0`.** *Rationale* : la lib `cryptography` est plus largement adoptée dans l'écosystème backend Python moderne (déjà transitivement présente via FastAPI/Starlette/JWT), maintenue par PyCA, expose une API de haut niveau (`Ed25519PrivateKey.from_private_bytes()`, `Ed25519PublicKey.verify()`) suffisante pour cette feature, et reste cohérente avec l'invariant « préférer librairies battle-tested » du `.cc-orchestrator.md`. `pynacl` reste une alternative valable post-MVP si HSM ou multi-curve nécessaires.

- **Q : Quelle bibliothèque pour la génération du QR code — `segno>=1.5` ou `qrcode>=7.4` ?**
  → **A : `segno>=1.5`.** *Rationale* : `segno` est plus léger (pas de dépendance Pillow obligatoire pour le PNG basique), produit des QR codes optimaux en taille (compression intelligente du contenu), et permet l'export direct PNG/SVG/EPS. Pour le MVP nous n'avons besoin que de PNG (`scale=10`, `error="M"` pour récupération moyenne), `segno` couvre ce besoin sans tirer Pillow. Pillow reste disponible si besoin post-MVP de manipulation d'image (logo overlay).

- **Q : Comment stocker la clé privée Ed25519 — variable d'environnement `.env`, fichier monté en volume, ou secret manager intégré ?**
  → **A : Variable d'environnement `ATTESTATION_PRIVATE_KEY_PEM` au format PEM (PKCS8 base64), chargée au démarrage de l'app via `core/config.py` puis mise en cache dans une singleton `signing.SigningKeyStore`. Un script `scripts/generate_attestation_keypair.py` one-shot génère le couple (clé privée + clé publique + `public_key_id="v1"`) et imprime sur stdout. La clé est stockée dans le secret manager de l'infra de production (jamais dans `.env` checké en git, jamais dans le repo). En dev, l'opérateur génère sa propre paire via le script et l'injecte dans son `.env` local.** *Rationale* : choix le plus simple pour le MVP, pas de dépendance externe (Vault/AWS Secrets Manager), aligné avec l'invariant #6 « jamais de secret hardcodé » et la pratique existante des autres clés (ex. `OPENROUTER_API_KEY`). Rotation post-MVP via `public_key_id` versionné (ATT-2026-XXX référencent la clé `v1`, futurs ATT-2027-XXX la `v2`). Documentation procédure dans `docs/attestations-and-verification.md` pour génération + injection sécurisée.

- **Q : Quelle stratégie face à l'énumération `/api/public/verify/{guess}` par un attaquant — distinguer 404 (UUID inexistant) de 200 « invalid » (UUID valide mais signature incorrecte) ?**
  → **A : Réponse uniforme HTTP 200 avec corps `{status: "invalid", message: "Cet identifiant d'attestation n'existe pas ou la signature est invalide", verified_at: <ISO>}` dans les deux cas (UUID inexistant et UUID valide mais signature corrompue), pour ne pas leak l'existence de l'identifiant. Aucune différenciation timing : le service charge systématiquement la table en SELECT (même quand UUID hors plage) et applique `time.sleep(0)` non utilisé (la latence DB normale couvre déjà la classe). Rate limiting strict 10 req/IP/min via FastAPI middleware (slowapi déjà candidate ou implémentation locale via cache LRU sur IP). Logs côté serveur tracent les tentatives d'énumération (>5 hits en 1 min depuis même IP → événement WARN). Pas de CAPTCHA pour le MVP (frottement utilisateur fund officer trop élevé).** *Rationale* : la fiche F08 explicite ce garde-fou risque #2. UUID v4 (entropie 122 bits) rend l'énumération aléatoire pratiquement impossible. Le rate limiting protège contre un attaquant déterminé. Le statut uniforme « invalid » couvre les deux cas (UUID inexistant ET signature corrompue) sans permettre de distinguer.

- **Q : Quels champs de l'attestation sont exposés sur la page publique `/verify/{id}` — vue minimale (juste le statut) ou métadonnées détaillées ?**
  → **A : Whitelist explicite côté serveur (jamais de sérialisation automatique du modèle SQLAlchemy) : pour `status='authentic'`, le serveur retourne `{status, attestation_id, attestation_type, valid_from, valid_until, issued_at, scores: {combined, solvability, green_impact, esg_global}, referentials: [{name, version, published_at}], pdf_hash_sha256, public_key_id, message}`. Pour `status='revoked'`, ajouter `{revoked_at, revoked_reason, revoked_by_role: 'pme' | 'admin'}` (PAS l'identité nominative du révocateur). Pour `status='expired'`, ajouter `{expired_since}`. Pour `status='invalid'`, uniquement `{status, message, verified_at}`. AUCUN nom d'entreprise, AUCUNE coordonnée, AUCUN détail des indicateurs ESG. Le fund officer doit comparer ces métadonnées avec le PDF reçu (le PDF lui-même contient l'attestation_id et le hash SHA-256 dans son pied de page).** *Rationale* : principe de minimisation RGPD (préparation F05). La page publique est consultée par un tiers non authentifié — la PME a confiance dans la transmission du PDF mais pas forcément dans la diffusion publique de son nom d'entreprise via une URL devinable.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Une PME génère une attestation signée et l'envoie à son banquier (Priorité : P1)

Une PME ivoirienne (déjà profilée et scorée crédit + ESG) souhaite répondre à une demande de financement formulée par son banquier ECOBANK. Elle accède à `/credit-score` (ou à la nouvelle page `/attestations`), clique sur « Générer une attestation vérifiable », sélectionne le type (combined : crédit + ESG), et confirme. Le backend génère le PDF avec QR code embarqué et URL de vérification visible, calcule le hash SHA-256, signe le payload `{attestation_id, scores, referentials, pdf_hash, valid_from, valid_until}` avec la clé privée Ed25519, persiste la `Attestation` row, et retourne au frontend l'URL de téléchargement + l'URL de vérification publique. La PME télécharge le PDF, le copie dans un email à son banquier, et copie également l'URL de vérification. Le banquier reçoit le PDF, scanne le QR avec son téléphone (ou clique sur l'URL), tombe sur la page `/verify/{id}` qui affiche **AUTHENTIQUE** avec les scores et les référentiels utilisés (versions). La conformité avec le PDF est vérifiable visuellement (l'attestation_id et le hash SHA-256 du PDF apparaissent dans le pied de page du PDF et sur la page publique).

**Pourquoi cette priorité** : c'est la promesse cœur de F08 et le scénario le plus visible auprès des partenaires fonds. Sans cette story, aucune des innovations de la fiche F08 (différenciateur produit n°2) n'est livrée. C'est P1 et bloquante pour le MVP global.

**Test indépendant** : se connecter en tant que PME ayant un score crédit + ESG calculé, accéder à `/attestations`, cliquer sur « Générer », confirmer le type `combined`, vérifier qu'une ligne `attestations` est créée avec `signature_ed25519`, `pdf_hash_sha256`, `qr_code_path`, `public_key_id='v1'`, `valid_from=now()`, `valid_until=now()+365d`, télécharger le PDF, ouvrir l'URL de vérification dans un onglet privé (no-auth), vérifier le statut **AUTHENTIQUE** avec les scores affichés.

**Scénarios d'acceptation** :

1. **Étant donné** une PME authentifiée avec un `CreditScore` et au moins une `EsgAssessment` finalisée existants, **quand** elle clique sur « Générer une attestation vérifiable » avec type `combined`, **alors** une ligne `attestations` est insérée avec tous les champs requis (`signature_ed25519` non vide, `pdf_hash_sha256` 64 chars hex, `qr_code_path` pointant vers un PNG existant, `verification_url` au format `https://esg-mefali.com/verify/{id}`, `valid_from=now()`, `valid_until=now()+365 jours`), l'audit log F03 contient une ligne `action='create' entity_type='attestations' source_of_change='manual'`.
2. **Étant donné** la PME télécharge le PDF généré, **quand** elle ouvre le PDF, **alors** elle voit le QR code, le verification_url en clair, l'identifiant `Attestation #ATT-{YYYY}-{NNNNN}` visible, le hash SHA-256 en pied de page, les scores (combined, solvability, green_impact, esg_global), les référentiels avec versions, et l'annexe « Sources et références » (auto-générée F01).
3. **Étant donné** un fund officer ouvre l'URL de vérification dans un onglet privé sans authentification, **quand** la page `/verify/{id}` charge, **alors** elle affiche un badge vert **AUTHENTIQUE** avec l'attestation_id, les scores, les référentiels (avec versions), `valid_from`, `valid_until`, le hash SHA-256, le `public_key_id='v1'`, sans aucune donnée nominative de l'entreprise (anonymisé sauf scores et structure ESG).
4. **Étant donné** un fund officer technique veut vérifier offline, **quand** il appelle `GET /api/public/attestation-public-key`, **alors** il reçoit la clé publique Ed25519 PEM + `public_key_id='v1'` et peut vérifier la signature en dehors de la plateforme (l'algorithme et le format canonique sont documentés dans `docs/attestations-and-verification.md`).

### User Story 2 — Un fund officer scanne le QR avec son téléphone et lit la page publique (Priorité : P1)

Un fund officer GCF reçoit un PDF d'attestation par email de la part d'une PME candidate. Il sort son téléphone, ouvre l'application appareil photo, scanne le QR code en haut du PDF. Le téléphone détecte l'URL `https://esg-mefali.com/verify/abc-1234-def-5678` et propose de l'ouvrir dans le navigateur. Le fund officer tape sur l'URL, atterrit sur une page mobile-first responsive (sans header user, sans sidebar, juste un header simple « ESG Mefali — Vérification d'attestation »). La page affiche immédiatement (en moins de 2 secondes sur 3G) un badge vert **AUTHENTIQUE** suivi des métadonnées (attestation_id, scores, référentiels avec versions, dates de validité, hash PDF). Le fund officer compare le hash affiché à l'écran avec celui imprimé sur le PDF (pied de page) — ils correspondent → l'attestation est authentique. Il peut copier l'attestation_id pour le coller dans son outil interne.

**Pourquoi cette priorité** : c'est l'expérience utilisateur côté fund officer, la plus visible aux yeux des partenaires fonds verts. La fiche F08 explicite « pas besoin de compte ». Sans cette page publique fonctionnelle, le scénario US1 ne sert à rien. P1 obligatoire.

**Test indépendant** : générer une attestation via US1, copier l'URL de vérification, ouvrir l'URL dans un onglet privé sans cookies (simule le fund officer non authentifié), vérifier que la page se charge sans redirection vers `/login`, que le badge **AUTHENTIQUE** s'affiche en vert, et que le hash SHA-256 affiché à l'écran est strictement égal au hash imprimé en pied de page du PDF téléchargé.

**Scénarios d'acceptation** :

1. **Étant donné** un fund officer non authentifié ouvrant `/verify/{attestation_id}` valide et non révoquée, **quand** la page se charge, **alors** elle affiche un badge vert **AUTHENTIQUE** avec attestation_id, scores combinés/solvabilité/impact/ESG, référentiels avec versions, `valid_from`/`valid_until` formatés en français, hash SHA-256 en monospace.
2. **Étant donné** un fund officer veut comparer le hash visible sur le PDF et celui de la page, **quand** il colle le hash dans le champ « Comparer avec votre PDF » de la page, **alors** la page affiche « Hash conforme — le PDF n'a pas été altéré » (vert) ou « Hash non conforme — le PDF reçu diffère de l'original » (rouge) selon comparaison stricte case-sensitive.
3. **Étant donné** la page publique `/verify/[id]`, **quand** elle est consultée depuis un téléphone (viewport 375x667), **alors** la mise en page est mobile-first (badge statut centré, scores affichés en cartes empilées, hash SHA-256 wrappé proprement), aucun débordement horizontal, et les boutons sont d'au moins 44x44 px (accessibilité tactile).
4. **Étant donné** un fund officer anglophone ouvrant la page publique avec `Accept-Language: en`, **quand** la page charge, **alors** le contenu est traduit en anglais (i18n FR/EN sur la page publique, infrastructure de traduction posée par F08 même si limitée à cette page pour le MVP).

### User Story 3 — La PME révoque une attestation devenue obsolète (Priorité : P1)

Une PME a généré une attestation il y a 3 mois. Depuis, son scoring crédit a évolué significativement (nouveau bilan, nouveau projet vert) et l'ancienne attestation reflète une réalité dépassée. Elle ne veut pas qu'un partenaire qui aurait reçu l'ancien PDF il y a quelque temps continue à le présenter comme authentique. Elle accède à la page `/attestations` (liste de ses attestations émises), trouve l'attestation concernée, clique sur « Révoquer », saisit la raison « Mise à jour majeure du profil financier » dans une modal de confirmation, valide. La ligne `attestations.revoked_at` est mise à jour avec `now()`, `revoked_reason` est persistée, `revoked_by_user_id` enregistre l'identité de la PME, l'audit log F03 trace l'action. Quelques minutes plus tard, le partenaire qui scanne le QR voit la page publique afficher **RÉVOQUÉE** (rouge) avec la date de révocation et la raison « Mise à jour majeure du profil ».

**Pourquoi cette priorité** : sans révocation par la PME elle-même, le système est non maintenable et la PME perd le contrôle de son attestation. C'est P1 car la révocation est une attente explicite des user stories de la fiche F08 et un garde-fou crédibilité.

**Test indépendant** : créer une attestation via US1, accéder à `/attestations`, cliquer sur « Révoquer » sur la ligne, saisir une raison, valider. Vérifier en base que `revoked_at`, `revoked_reason`, `revoked_by_user_id` sont peuplés. Recharger `/verify/{id}` (en onglet privé) et vérifier que la page affiche **RÉVOQUÉE** avec la date et la raison.

**Scénarios d'acceptation** :

1. **Étant donné** une PME authentifiée avec une attestation `authentic` lui appartenant, **quand** elle clique sur « Révoquer » et saisit une raison de minimum 10 caractères, **alors** la ligne est mise à jour avec `revoked_at=now()`, `revoked_reason=<raison>`, `revoked_by_user_id=<user.id>`, l'audit log F03 contient `action='revoke' entity_type='attestations' source_of_change='manual'`, et un toast UI confirme « Attestation révoquée avec succès ».
2. **Étant donné** une attestation révoquée, **quand** quelqu'un ouvre `/verify/{id}` sans authentification, **alors** la page affiche un badge rouge **RÉVOQUÉE** avec `revoked_at` formaté en français, `revoked_reason`, et `revoked_by_role='pme'` (sans le nom de la personne).
3. **Étant donné** une PME tente de révoquer une attestation appartenant à un autre tenant (cross-tenant), **quand** la requête arrive au serveur, **alors** elle reçoit HTTP 404 (pas d'information sur l'existence de l'attestation appartenant à autrui — RLS PostgreSQL F02 bloque la lecture, le tool LangChain `revoke_attestation` retourne `{ok: false, error: "Attestation introuvable"}`).
4. **Étant donné** une PME tente de révoquer une attestation déjà révoquée, **quand** elle confirme, **alors** la requête est rejetée avec HTTP 409 (`already_revoked`) et le frontend affiche « Cette attestation est déjà révoquée » sans modifier l'état en base.

### User Story 4 — Un admin révoque une attestation suite à un signalement de fraude (Priorité : P2)

Un admin ESG Mefali (rôle Admin F02) reçoit une alerte de fraude potentielle via un canal externe (email d'un fund officer signalant une discordance entre le PDF transmis et les données réelles). Il accède à la page d'administration `/admin/attestations`, recherche l'attestation par son identifiant, examine les métadonnées (PME concernée, type, dates, scores), clique sur « Révoquer » avec une raison obligatoire (« Suspicion de fraude — investigation en cours »), valide. La ligne est mise à jour avec `revoked_by_user_id=<admin.id>`, l'audit log trace l'action avec `actor_role='admin'`. La page publique `/verify/{id}` affiche désormais **RÉVOQUÉE** avec `revoked_by_role='admin'` (sans nom). La PME concernée reçoit un événement audit log F03 avec un libellé clair.

**Pourquoi cette priorité** : nécessaire pour un canal d'incident en production (fraude, erreur de calcul majeure) mais P2 car le MVP peut survivre sans cette interface admin si la révocation est faite manuellement en base par un opérateur (post-MVP automatisé). La révocation par admin est une exigence de la fiche F08 mais pas bloquante pour la première vague d'utilisateurs.

**Test indépendant** : créer une attestation via US1, se connecter en tant qu'admin (F02 fournit le rôle Admin via `get_current_admin`), accéder à `/admin/attestations`, trouver l'attestation, révoquer avec une raison, vérifier l'audit log et la page publique.

**Scénarios d'acceptation** :

1. **Étant donné** un admin authentifié, **quand** il accède à `/admin/attestations`, **alors** il voit la liste de TOUTES les attestations (cross-tenant, RLS contournée par le rôle Admin F02), avec colonnes `attestation_id`, `account_id`, `type`, `status`, `valid_from`, `valid_until`, `revoked_at`.
2. **Étant donné** un admin clique sur « Révoquer » pour une attestation, **quand** il saisit une raison ≥ 10 caractères et confirme, **alors** la ligne est révoquée, l'audit log F03 contient `actor_role='admin'`, `source_of_change='manual'`, et la page publique reflète immédiatement le statut **RÉVOQUÉE**.
3. **Étant donné** un user PME (non admin) tente d'accéder à `/admin/attestations`, **quand** la page charge, **alors** le middleware redirige vers `/dashboard` (ou affiche 403) sans révéler la liste des attestations (sécurité défense en profondeur).

### User Story 5 — Le LLM génère une attestation depuis le chat (refactor du tool placeholder) (Priorité : P2)

Une PME demande à l'assistant IA dans le chat : « Génère-moi une attestation pour mon score crédit. » Le LLM identifie l'intention, vérifie qu'un `CreditScore` existe pour la PME, appelle le tool LangChain `generate_credit_certificate(attestation_type='credit_score')`. Le tool n'est plus un placeholder — il appelle réellement le service `attestations.service.generate_attestation`. La génération prend ~3 secondes (PDF + signature + QR), pendant lesquelles le LLM affiche un message de patience. Une fois terminé, le tool retourne `{ok, attestation_id, verification_url, pdf_path}`. Le LLM répond : « Votre attestation est prête. Vous pouvez la télécharger ici : [lien] ou la vérifier publiquement à : `https://esg-mefali.com/verify/abc-1234`. » L'utilisateur peut copier l'URL en presse-papier (action UI).

**Pourquoi cette priorité** : le LLM est un canal de génération d'attestation aussi naturel que l'UI directe, et c'est cohérent avec le principe constitutionnel III (Conversation-Driven UX). P2 car les utilisateurs peuvent passer par l'UI directe (US1) si nécessaire ; la conversation enrichit l'expérience.

**Test indépendant** : démarrer une conversation avec « Génère-moi une attestation pour mon score crédit », observer que le LLM appelle le tool `generate_credit_certificate` (visible via SSE event `tool_call_start`), vérifier que la ligne `attestations` est créée, que le tool retourne l'objet attendu, et que le LLM affiche le `verification_url` dans sa réponse texte. Comparer avec l'ancien comportement placeholder (test de non-régression : avant F08, le tool retournait une URL fictive ; après F08, l'URL doit pointer vers une vraie attestation).

**Scénarios d'acceptation** :

1. **Étant donné** une PME en chat avec un `CreditScore` existant, **quand** elle demande « Génère mon attestation », **alors** le LLM appelle `generate_credit_certificate(attestation_type='credit_score')`, le service crée la `Attestation` row réelle, et le tool retourne `{ok: true, attestation_id, verification_url, pdf_path}`.
2. **Étant donné** le tool `generate_credit_certificate` est appelé sans `CreditScore` existant, **quand** le service tente de générer, **alors** il retourne `{ok: false, error: "Aucun score crédit calculé. Veuillez d'abord finaliser le scoring crédit."}` et le LLM relaie le message à l'utilisateur.
3. **Étant donné** la génération PDF prend > 5 secondes (cas exceptionnel), **quand** le tool est appelé, **alors** un timeout de 10 secondes est appliqué et le tool retourne `{ok: false, error: "Génération en cours, vérifiez votre liste d'attestations dans quelques instants"}` (pas de blocage indéfini du chat).

### User Story 6 — Une attestation expire automatiquement après 1 an (Priorité : P3)

Une PME a généré une attestation il y a 13 mois. Aujourd'hui, un partenaire scanne le QR. Le service backend détecte que `valid_until < now()` et retourne `status='expired'`. La page publique affiche un badge orange **EXPIRÉE** avec la date d'expiration et un message « Cette attestation a expiré. Demandez à la PME de générer une nouvelle attestation. » La PME, de son côté, voit dans `/attestations` un badge « expirée » sur cette ligne et peut générer une nouvelle attestation (qui sera une nouvelle ligne, l'ancienne reste en base pour audit).

**Pourquoi cette priorité** : l'expiration automatique est une garantie de fraîcheur du score (le profil PME évolue). P3 car le MVP peut tolérer des attestations long-vivantes, mais cette story est nécessaire pour la cohérence produit.

**Test indépendant** : créer une attestation, manipuler artificiellement `valid_until` à `now() - 1 day` en base, recharger `/verify/{id}` et vérifier le statut **EXPIRÉE** affiché.

**Scénarios d'acceptation** :

1. **Étant donné** une attestation avec `valid_until < now()` non révoquée, **quand** quelqu'un ouvre `/verify/{id}` sans authentification, **alors** la page retourne `status='expired'` avec `expired_since=<date d'expiration formatée>` et affiche un badge orange.
2. **Étant donné** une attestation expirée, **quand** la PME tente de la révoquer, **alors** la requête est acceptée (idempotente sur révocation tardive) MAIS le statut côté page publique reste `expired` (priorité au statut le plus contraignant : `revoked` > `expired` > `authentic`).
3. **Étant donné** une attestation qui expirera dans moins de 30 jours, **quand** la PME consulte `/attestations`, **alors** un badge « expire bientôt — il vous reste N jours » est rendu et un bouton « Renouveler » est proposé (créera une nouvelle attestation en parallèle, l'ancienne expirera naturellement).

### Edge Cases

- **Tentative d'altération du PDF** : un attaquant modifie le PDF (ajoute du texte, change un score) avec un éditeur PDF. Le hash SHA-256 ne correspond plus à celui stocké en base. Le fund officer scanne le QR : la page publique retourne le hash original, l'utilisateur compare → discordance détectée → l'attestation reste affichée comme **AUTHENTIQUE** (la signature de la ligne en base est correcte) MAIS le hash visible ne correspond pas au hash du PDF reçu. La page invite explicitement « Vérifiez que le hash visible ci-dessus correspond au hash imprimé en pied de page de votre PDF ». Pas de protection automatique côté serveur (il ne peut pas savoir quel PDF l'utilisateur a reçu).
- **Tentative de génération par utilisateur PME malveillant** : un user PME altère sa requête API pour injecter un score différent. Comme le service backend recalcule le payload depuis la base (`scores` chargés via `account_id` + dernier `CreditScore`/`EsgAssessment` actuel), il ne peut PAS injecter un score arbitraire — la signature s'applique sur le payload côté serveur, jamais sur des données utilisateur.
- **Tentative d'énumération `/api/public/verify/{guess}`** : rate limiting 10 req/IP/min, statut uniforme `invalid` pour UUID inexistant et signature corrompue, pas de différenciation timing.
- **Attestation révoquée puis tentative de re-génération avec mêmes données** : autorisée (rien n'empêche une PME de révoquer puis re-générer une attestation pour le même score). La nouvelle attestation a un `attestation_id` différent et un `valid_from` plus récent. L'ancienne reste en base pour audit, marquée révoquée.
- **PDF généré mais persistance échoue** : transactional avec rollback. Le PDF temporaire (`/tmp/attestations/draft-{uuid}.pdf`) est supprimé. Aucune ligne `attestations` n'est créée. L'utilisateur reçoit une erreur explicite et peut réessayer.
- **Clé privée Ed25519 absente au démarrage** : l'app refuse de démarrer (fail-fast en production) avec un log explicite « ATTESTATION_PRIVATE_KEY_PEM manquant — exécutez scripts/generate_attestation_keypair.py ». En tests : la clé privée est générée à la volée par une fixture pytest.
- **Concurrence : deux générations simultanées par la même PME** : autorisée. Chaque génération crée une nouvelle ligne `attestations` avec un UUID v4 différent. Pas d'unicité par PME (une PME peut avoir plusieurs attestations vivantes en parallèle, ex. crédit + ESG + combined).
- **Attestation pour PME multi-tenant** : multi-tenant strict (F02). `account_id` NOT NULL sur `attestations`. RLS PostgreSQL bloque la lecture cross-tenant (sauf rôle Admin). L'admin peut révoquer cross-tenant.
- **Render PDF en mode preview avant génération** : hors-scope MVP. Le PDF est généré à la confirmation finale, sans prévisualisation. Post-MVP : preview HTML avant génération PDF définitive.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001** : Le système DOIT créer une nouvelle ligne `attestations` à chaque génération avec tous les champs requis : `id` UUID v4, `account_id`, `user_id`, `attestation_type` (enum `credit_score|esg_assessment|combined`), `payload` (JSONB snapshot des scores et référentiels avec versions), `referential_snapshot` (JSONB), `pdf_path` (chemin local), `pdf_hash_sha256` (64 chars hex), `signature_ed25519` (base64), `public_key_id` (chaîne, défaut `"v1"`), `qr_code_path` (chemin PNG), `valid_from` (timestamp), `valid_until` (timestamp = valid_from + 365 jours par défaut), `revoked_at`/`revoked_reason`/`revoked_by_user_id` (NULLs initialement), `verification_url` (chaîne), `created_at`, `updated_at`.
- **FR-002** : La signature Ed25519 DOIT être générée serveur-side avec une clé privée chargée depuis `ATTESTATION_PRIVATE_KEY_PEM` (env var). La signature s'applique au format canonique JSON (`json.dumps(..., sort_keys=True, separators=(",", ":"))`) du payload incluant `attestation_id`, `scores`, `referential_snapshot`, `pdf_hash_sha256`, `valid_from`, `valid_until`. La signature est encodée en base64 dans `signature_ed25519`.
- **FR-003** : Le hash SHA-256 du PDF généré DOIT être calculé après écriture finale du fichier et persisté dans `pdf_hash_sha256` (64 chars hex lowercase). Toute altération ultérieure du PDF rend le hash incohérent (la PME peut comparer le hash visible sur la page publique avec celui imprimé en pied de page du PDF).
- **FR-004** : Le QR code DOIT être généré au format PNG via `segno` avec `error="M"` (récupération moyenne) et `scale=10` (300x300 px environ), pointant vers `verification_url` au format `{ATTESTATION_VERIFICATION_BASE_URL}/verify/{attestation_id}` (base URL configurable via env var, défaut `https://esg-mefali.com`). Le PNG est stocké dans `/uploads/attestations/qr/{attestation_id}.png`.
- **FR-005** : Le PDF généré DOIT contenir, en plus du contenu existant : le QR code embarqué (image PNG), le `verification_url` en clair (texte sous le QR), l'identifiant visible (`Attestation #ATT-{YYYY}-{NNNNN}` calculé à partir de l'année et d'un compteur séquentiel scopé `account_id`), le hash SHA-256 du PDF en pied de page (en hexa monospace), les référentiels utilisés avec versions (« Évalué selon ESG Mefali v1.2 du 15/03/2026 + GCF Investment Framework v2.3 »), une annexe « Sources et références » auto-générée (F01), et le disclaimer juridique.
- **FR-006** : Un endpoint `GET /api/public/verify/{attestation_id}` DOIT être accessible SANS authentification (le router est monté AVANT le middleware d'auth dans `app/main.py`). Il retourne un statut whitelisté : `authentic` (signature valide + non révoquée + non expirée), `revoked` (révoquée), `expired` (`valid_until < now()`), `invalid` (UUID inexistant OU signature corrompue, pas de différenciation). La réponse expose UNIQUEMENT les métadonnées non sensibles (jamais le nom d'entreprise, les coordonnées, ou le détail des indicateurs).
- **FR-007** : Un endpoint `GET /api/public/attestation-public-key` DOIT exposer la clé publique Ed25519 (PEM) et le `public_key_id` actif, permettant à un fund officer technique de vérifier la signature offline.
- **FR-008** : Un endpoint authentifié `POST /api/attestations/{id}/revoke` DOIT permettre à la PME propriétaire de révoquer son attestation. Le payload requiert `reason` (string, min 10 chars). RLS bloque toute révocation cross-tenant. L'opération met à jour `revoked_at`, `revoked_reason`, `revoked_by_user_id` et émet un événement audit log F03 (`action='revoke'`, `source_of_change='manual'`).
- **FR-009** : Un endpoint admin `POST /api/admin/attestations/{id}/revoke` DOIT permettre à un user de rôle Admin (F02 `get_current_admin`) de révoquer toute attestation, cross-tenant. L'audit log F03 enregistre `actor_role='admin'`.
- **FR-010** : Le tool LangChain `generate_credit_certificate` (renommé en `generate_attestation` ou conservé pour rétrocompatibilité) DOIT appeler réellement `attestations.service.generate_attestation` avec le user courant et l'`attestation_type` demandé. Il retourne `{ok, attestation_id, verification_url, pdf_path}` ou `{ok: false, error}` en cas d'échec. Plus de placeholder.
- **FR-011** : La page Vue `/verify/[id]` DOIT être accessible sans authentification (le middleware `auth.global.ts` autorise explicitement `/verify/` et `/legal/`). Le layout `public.vue` (réutilisable) ne contient ni sidebar, ni header user, juste un header simple « ESG Mefali — Vérification d'attestation » et un footer minimaliste. La page est mobile-first responsive (viewport 375x667 OK, boutons ≥ 44x44 px).
- **FR-012** : La page `/verify/[id]` DOIT afficher 4 statuts visuels distincts : **AUTHENTIQUE** (badge vert), **RÉVOQUÉE** (badge rouge avec date et raison), **EXPIRÉE** (badge orange avec date d'expiration), **INVALIDE** (badge rouge avec message neutre). Pour chaque statut, les métadonnées affichées sont strictement whitelisted côté serveur (jamais de fuite par sérialisation automatique).
- **FR-013** : La PME DOIT pouvoir lister ses attestations émises sur `/attestations` (page Vue authentifiée) avec colonnes : attestation_id, type, statut visuel, valid_from/valid_until, actions (télécharger PDF, copier URL de vérification, révoquer si non révoquée). Tri par `created_at DESC`.
- **FR-014** : L'admin DOIT pouvoir lister TOUTES les attestations sur `/admin/attestations` (page Vue admin) avec colonnes complémentaires (account_id, user_id) pour audit. Filtrage par statut, par tenant, par période.
- **FR-015** : Le système DOIT appliquer un rate limiting de 10 req/IP/min sur `GET /api/public/verify/{id}` pour empêcher l'énumération. Au-delà du seuil, retourne HTTP 429.
- **FR-016** : Le système DOIT logger toute tentative d'énumération suspecte (>5 hits depuis même IP en 1 min sur `/api/public/verify/`) au niveau WARN avec adresse IP et user-agent, sans persistence en BDD (logs only).
- **FR-017** : Le système DOIT exposer une page publique mobile-first responsive avec i18n FR/EN basique (au minimum les libellés `AUTHENTIQUE/AUTHENTIC`, `RÉVOQUÉE/REVOKED`, `EXPIRÉE/EXPIRED`, `INVALIDE/INVALID`, et les boutons de comparaison hash). Détection via `Accept-Language` ou `?lang=en`.
- **FR-018** : Toute mutation sur `attestations` (création, révocation par PME ou admin) DOIT générer une entrée audit log F03 avec `entity_type='attestations'`, `action ∈ {create, revoke}`, `source_of_change ∈ {manual, llm}` (LLM pour génération via tool LangChain), `actor_role ∈ {pme, admin}`.
- **FR-019** : Si le tool LLM `generate_credit_certificate` cite un score (target_amount estimé, expected_impact_tco2e, etc.) dans la conversation, ce chiffre DOIT être lié à une `Source` `verified` via `cite_source` (F01) ou marqué `unsourced` via `flag_unsourced`. Le validator F01 reste actif sur le module attestations.
- **FR-020** : Le système DOIT permettre la rotation de clé Ed25519 post-MVP via `public_key_id` versionné. Pour le MVP, `public_key_id="v1"` constant. Une migration future (post-MVP) introduira une table `attestation_keys` pour stocker l'historique des clés publiques.

### Key Entities

- **Attestation** : représente un acte de certification ponctuel d'un score (crédit, ESG, ou combiné), signé numériquement Ed25519, lié à un `account_id` (multi-tenant F02), Auditable (F03). Snapshot complet du payload (scores, référentiels avec versions) au moment de la génération. Cycle de vie : `authentic → revoked` (par PME ou admin) ou `authentic → expired` (automatique après 1 an). Champs principaux : `id` (UUID v4 exposé publiquement comme attestation_id), `account_id`, `user_id`, `attestation_type` (enum), `payload` (JSONB), `referential_snapshot` (JSONB), `pdf_path`, `pdf_hash_sha256`, `signature_ed25519`, `public_key_id`, `qr_code_path`, `valid_from`, `valid_until`, `revoked_at`, `revoked_reason`, `revoked_by_user_id`, `verification_url`, `created_at`, `updated_at`.
- **VerificationResult** (DTO public) : objet retourné par l'endpoint `/api/public/verify/{id}`, structure différente selon le statut. Champs garantis : `status` (enum `authentic|revoked|expired|invalid`), `verified_at` (timestamp serveur de la vérification), `message` (libellé i18n FR/EN). Champs supplémentaires conditionnels : pour `authentic` (`attestation_id`, `attestation_type`, `valid_from`, `valid_until`, `issued_at`, `scores`, `referentials`, `pdf_hash_sha256`, `public_key_id`) ; pour `revoked` (`revoked_at`, `revoked_reason`, `revoked_by_role`) ; pour `expired` (`expired_since`).
- **AttestationKey (post-MVP)** : table satellite pour stocker l'historique des clés publiques Ed25519 utilisées pour signature, permettant la rotation. Champs : `public_key_id` (PK), `public_key_pem`, `algorithm='ed25519'`, `active` (bool), `rotated_at` (NULL tant qu'active). Hors-scope MVP — pour le MVP, une seule clé `v1` est en service.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001** : 100 % des attestations générées DOIVENT être vérifiables via `GET /api/public/verify/{id}` en moins de 1 seconde sur 4G (latence cumulée backend + frontend ≤ 2 secondes p95).
- **SC-002** : 100 % des PDF générés DOIVENT contenir un QR code scannable (testé sur smartphone iOS et Android, application appareil photo native), un identifiant visible `Attestation #ATT-YYYY-NNNNN`, et le hash SHA-256 en pied de page.
- **SC-003** : 100 % des tentatives de scanner d'un QR aboutissent à une page publique chargée en moins de 2 secondes sur 3G simulée (pas de redirect vers login, pas de prompt de cookie).
- **SC-004** : Sur un échantillon de 10 attestations générées avec couples (signature, payload), la vérification offline avec la clé publique Ed25519 retourne `valid` à 100 % (testé via script Python externe utilisant `cryptography.hazmat.primitives.asymmetric.ed25519.Ed25519PublicKey.verify`).
- **SC-005** : Couverture de test ≥ 85 % sur les modules `app/modules/attestations/signing.py`, `app/modules/attestations/qr.py`, et `app/modules/attestations/service.py`. Couverture globale ≥ 80 % sur l'ensemble de F08 (backend + frontend).
- **SC-006** : Aucune fuite d'information entre les 4 statuts publics (`authentic`, `revoked`, `expired`, `invalid`) : la latence p50/p95 est statistiquement indistinguable (test de Mann-Whitney sur 100 requêtes par classe, p > 0.1).
- **SC-007** : Le rate limiting bloque > 95 % des tentatives d'énumération (test : 100 requêtes en 30 secondes depuis même IP → ≥ 95 % retournent HTTP 429).
- **SC-008** : Les 5 scénarios E2E Playwright (`F08-attestation-verifiable-ed25519.spec.ts`) s'exécutent en < 60 secondes au total et passent à 100 % en CI sur 5 runs consécutifs (zéro flaky).

## Assumptions

- **Cibles utilisateurs** : la page publique `/verify/[id]` est principalement consultée par des fund officers / banquiers ayant un téléphone moderne (iOS 14+ ou Android 10+) avec lecture native de QR code via l'application appareil photo. Pas d'accessibilité spécifique pour téléphones à clapet (hors-scope MVP).
- **Connectivité** : la page publique est consultable en 3G (zones rurales africaines). Le poids de la page DOIT être < 200 ko gzip (pas de hero images, pas de framework lourd côté client).
- **Confiance dans la chaîne** : on suppose que la clé privée Ed25519 reste secrète et stockée dans un secret manager en production. Aucune compromission n'est anticipée pendant le MVP. Procédure de rotation documentée mais non automatisée.
- **Rétention des PDF/QR sur disque** : conservés indéfiniment dans `/uploads/attestations/`. Politique de purge post-MVP (par exemple : supprimer les PDF physiques 5 ans après expiration, mais conserver la ligne `attestations` en base pour audit). Hors-scope MVP.
- **Performance génération** : la génération PDF + signature + QR prend ~3 secondes en moyenne. Acceptable côté UX (l'utilisateur a un message de patience). Bascule en background task FastAPI si > 10 secondes.
- **Volumétrie attendue** : MVP ~100-1000 attestations par mois (~1 PDF de 2-3 Mo par attestation, ~3 ko par QR PNG). Stockage local suffisant (estimation 30 Go sur 1 an avec marges).
- **i18n** : FR par défaut. EN sur la page publique uniquement (libellés statiques + 4 messages de statut). Pas d'i18n complète sur le reste de l'app pour cette feature.
- **Compatibilité navigateurs** : Chrome 90+, Safari 14+, Firefox 90+, Edge 90+ sur la page publique. Aucun support IE.
- **Intégration F15 (FundApplication)** : la possibilité de joindre une `Attestation` existante à une `FundApplication` lors de la génération d'un dossier (Module 3.3) est mentionnée comme pré-câblage souhaitable mais sera réellement implémentée par F15 (hors-scope F08, juste une FK nullable `fund_applications.attestation_id` non créée maintenant — sera ajoutée par F15).
- **Watermark anti-altération** : hors-scope MVP. Le hash SHA-256 visible sur le PDF + QR vers page publique constituent la défense en profondeur.
- **Ancrage blockchain / notarization tiers de confiance** : hors-scope MVP, pertinent post-MVP si demande commerciale.
- **Wallet PME multi-attestations** : hors-scope MVP. Une PME peut avoir plusieurs attestations en parallèle dans `/attestations`, mais pas de wallet dédié.
