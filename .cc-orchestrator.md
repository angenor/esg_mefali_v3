## INVARIANTS PROJET — NON NÉGOCIABLES

Ces règles sont issues de la lecture des 24 fiches et du `CLAUDE.md`. **Tout sous-agent
qui les viole DOIT être renvoyé en correction.**

1. **Sourçage obligatoire** (F01 introduit) : à partir du moment où F01 est `done`, tout chiffre,
   score, critère, formule, seuil, facteur d'émission affiché DOIT être lié à une `Source`
   `verified` via le tool LangChain `cite_source(source_id)`. Le validator backend rejette
   les réponses LLM contenant des chiffres sans citation. AVANT F01 : marquer les nouveaux
   chiffres avec `# TODO(F01): cite_source` pour migration ultérieure.

2. **Multi-tenant strict** (F02 introduit) : à partir du moment où F02 est `done`, toute
   nouvelle table métier DOIT avoir `account_id: UUID FK accounts.id NOT NULL` + RLS policy.
   AVANT F02 : pas de nouvelle table métier sans `user_id` au minimum, marquer avec
   `# TODO(F02): account_id` pour migration ultérieure.

3. **Audit log append-only** (F03 introduit) : à partir du moment où F03 est `done`, toute
   mutation métier DOIT passer par un service décoré du mixin `Auditable`. Pas de `db.commit()`
   direct dans les tools LangChain. AVANT F03 : conserver l'usage des services existants.

4. **Versioning + Money typed** (F04 introduit) : à partir du moment où F04 est `done`, tout
   nouveau champ monétaire DOIT être `(amount: Numeric(20,2), currency: Char(3))` paire et
   utiliser `Money` Pydantic dans les schémas. Plus de `*_xof` simples. AVANT F04 : OK de
   garder mono-devise XOF, marquer `# TODO(F04): Money typed`.

5. **RGPD consentements** (F05 introduit) : à partir du moment où F05 est `done`, tout
   traitement non-essentiel (Mobile Money, photos IA, données publiques, attestation) DOIT
   appeler `require_consent(account_id, type)` au runtime. Pas de collecte sans consent.

6. **Aucun secret hardcodé** : URLs OpenRouter, API keys, mots de passe, clés Ed25519 → env
   vars uniquement (`backend/app/core/config.py`).

7. **Aucun tool LLM ne mute le catalogue** : Funds, Intermediaries, Sources, Skills, Indicators,
   Referentials, EmissionFactors, SimulationFactors, Templates → réservés Admin (F09).

8. **Dark mode obligatoire** : tout nouveau composant Vue DOIT inclure les variantes `dark:`
   Tailwind. Voir `CLAUDE.md` section "Dark Mode (OBLIGATOIRE)".

9. **Réutilisabilité composants** : avant de créer un composant Vue, vérifier
   `frontend/app/components/ui/` ; si pattern apparaît > 2 fois, extraire.

10. **Français avec accents** (é, è, ê, à, ç, ù) dans les contenus utilisateur, prompts,
    documentation. Code en anglais.

11. **Tests E2E exécutables** : tous les tests E2E doivent être Playwright (`frontend/tests/e2e/`)
    et exécutables via `cd frontend && npx playwright test`. Pas de listes de tests manuels.

12. **Couverture tests ≥ 80 %** : backend via pytest-cov, frontend via vitest --coverage.

## BOUCLE D'ORCHESTRATION

```
1. Lire .cc-deps.json (deps, done, in_progress, blocked)
2. ready = { f : f ∉ done ∪ in_progress ∪ blocked  ET  deps(f) ⊆ done }
3. Lancer en // jusqu'à _max_parallel=6 : Phase A simultanées (Task subagent_type=general-purpose)
4. À réception du rapport JSON A → spawn Phase B (FRAIS)
5. À réception du rapport JSON B → spawn Phase B' (FRAIS, subagent_type=e2e-runner)
6. À réception du rapport JSON B' (passing) → COMMIT/PUSH/PR/MERGE EN SÉRIE (orchestrateur)
7. Mettre à jour .cc-deps.json : in_progress → done, mettre à jour .cc-queue.md
8. Logger dans .cc-runtime/logs/orchestration.log
9. Recalculer ready (étape 2), boucler
```

## COMMIT / PUSH / PR / MERGE (ORCHESTRATEUR, EN SÉRIE)

```bash
git checkout feat/F<NN>-<slug>
git add -A
git commit -m "feat(F<NN>): <titre court tiré de spec.md>

<corps optionnel : détails majeurs>"
git push -u origin feat/F<NN>-<slug>
gh pr create --base main --fill --label "auto-generated,F<NN>"
gh pr merge <num> --squash --delete-branch
git checkout main && git pull --ff-only
```

Convention commit : `<type>(F<NN>): <description>` (Conventional Commits).
Types : `feat`, `fix`, `refactor`, `docs`, `test`, `chore`, `perf`, `ci`.

### MAJ état (orchestrateur uniquement)
- `.cc-deps.json` : transition `in_progress` → `done`
- `.cc-queue.md` : `[ ]` → `[x]` (ou `[!]` si blocked)
- `.cc-runtime/logs/orchestration.log` :
  ```
  <ISO timestamp> | F<NN>-<slug> | DONE | PR=<url> | merged | e2e=<n>/<total> | coverage=<pct>
  ```

## CONFLITS MERGE

`gh pr merge` échoue → checkout branche, sous-agent FRAIS "rebase + résoudre sans casser
tests", re-push, retry. 2 échecs → marquer `blocked`.

## GARDE-FOUS GLOBAUX

1. **Jamais de commit sur `main` directement.**
2. **Jamais de** `--no-verify`, `--force-push`, `git reset --hard` (sauf demande explicite user, absent ici).
3. **Jamais de secret hardcodé.** Vérifier avec `grep -rE '(api_key|secret|password|token)\s*=\s*["\047][A-Za-z0-9]' backend/ frontend/` avant chaque commit.
4. **Jamais de suppression hors périmètre** (pas de `rm -rf`, pas de drop table, pas de purge).
5. **Jamais déranger l'utilisateur** (mode autonomie totale).
6. **Toujours commit avant merge.**
7. **Conventional commits** : `<type>(F<NN>): <description>`.
8. **Toujours sous-agent FRAIS** pour A, B, B' (pas de réutilisation pour éviter contamination contexte).
9. **Sérialiser les migrations** : UNE seule migration Alembic en flight max. Si sous-agent
   B veut créer migration alors qu'une autre est en cours d'écriture sur une autre branche,
   signaler `zone_conflict`.
10. **Mettre à jour `.cc-deps.json` après chaque transition** (atomicité par l'orchestrateur).

## DÉCISIONS PAR DÉFAUT POUR `/speckit.clarify`

Si une question SpecKit n'a pas de "Recommended" :

| Question type | Décision par défaut | Rationale |
|---|---|---|
| Migration BDD : drop colonne ou conserver legacy ? | Conserver legacy `_deprecated` 2 sprints | Réversibilité |
